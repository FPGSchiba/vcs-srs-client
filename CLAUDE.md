## What this project is

The **Vanguard Communications System (VCS) client** — a desktop voice-comms application for the Star Citizen community using the SRS-style radio paradigm. Multi-window, distributed-voice-aware, cross-platform (Windows / macOS / Linux). Stack:

- **Frontend:** Wails v3 + React 18 + TypeScript + Vite + Tailwind + shadcn/ui + Zustand + TanStack Query/Router. Per-window Vite entries (one bundle per OS window).
- **Backend:** Go (`github.com/FPGSchiba/vcs-srs-client`), `log/slog` logger, `google.golang.org/grpc` for control, custom UDP for voice (Phase 5+), `malgo` for audio (Phase 4+).
- **Wire contract:** `srs.proto` at repo root. Bindings generated into `srspb/` (gitignored, regenerated by CI).

Foundational decisions are locked in the design spec — **do not re-litigate the tech stack** in new sessions. If something needs to change, update the spec and surface it to the user before touching code.

## Mandatory workflow — starting any new phase

> **When the user says any of: "let's start Phase N", "begin the next phase", "pick up Phase N", "let's do Phase N", or any equivalent — you MUST invoke the `superpowers:brainstorming` skill before writing any code, scaffolding, or implementation plan.**

This is non-negotiable. Each phase gets its own design doc derived from the master spec before implementation. The brainstorming flow will:

1. Read the master design spec and the relevant `ROADMAP.md` row.
2. Ask clarifying questions and propose 2 – 3 approaches.
3. Present the phase design in sections for approval.
4. Write the phase design doc to `docs/superpowers/specs/YYYY-MM-DD-vcs-client-phase-N-<theme>-design.md`.
5. Commit it.
6. Hand off to `superpowers:writing-plans` only after the user approves the spec.

If `superpowers:brainstorming` is unavailable in the current environment, **stop and report it** rather than skipping the step.

## Mandatory workflow — executing an approved plan

When a phase has an approved design doc *and* an approved implementation plan (`docs/superpowers/plans/...`):

- Use `superpowers:executing-plans` or `superpowers:subagent-driven-development` to drive execution.
- Follow `superpowers:test-driven-development`: tests first, implementation second.
- Run `superpowers:verification-before-completion` before claiming any task done.
- Use `superpowers:requesting-code-review` before merge.

## What requires the user's explicit approval

- Adding or changing a dependency (Go or npm).
- Changing the proto file. (`docs/PROTO_GAPS.md` is the queue — propose changes there first.)
- Anything that crosses a phase boundary. If you find yourself wanting to implement something from Phase N+1 while working on Phase N, stop and ask.
- Any deviation from a decided architecture point in the master spec.
- Destructive git operations, dependency downgrades, modifying CI/CD pipelines.
